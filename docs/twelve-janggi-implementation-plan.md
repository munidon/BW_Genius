# 십이장기 구현 계획

## 목표
1. 같은 사이트 안에 `십이장기`를 새 보드게임으로 추가한다.
2. 방 생성, 입장, 준비, 시작, 나가기 흐름은 기존 흑과백과 비슷하게 유지한다.
3. 게임 전용 데이터는 흑과백과 충돌하지 않도록 분리 관리한다.
4. UI는 흑과백의 구조를 재사용하되, 전체 분위기는 녹색 계열로 새로 구성한다.

## 분리 전략
1. 인증은 기존 `auth.users`를 그대로 사용한다.
2. 현재 공용 닉네임 흐름은 `components/auth-provider.tsx` 기준으로 `bw_profiles`에 묶여 있으므로, v1에서는 로그인과 닉네임은 공용으로 유지한다.
3. 십이장기의 룸, 게임 상태, 수순 로그, 전적 집계는 흑과백과 완전히 분리한다.
4. 우선안은 `twelve_janggi` 전용 스키마를 두는 방식이다.
5. Supabase 노출 설정 때문에 전용 스키마 운용이 번거로우면 차선안으로 `public` 스키마 안에 `tj_` 접두사 테이블과 RPC를 둔다.
6. 어떤 안을 쓰더라도 `bw_*` 테이블, RPC, Realtime 채널과 공유하지 않는다.

## 권장 데이터 구조
1. `rooms`
   - 방 코드, 호스트/게스트, 준비 상태, 대기/플레이/종료 상태, 승자, 갱신 시각 관리
2. `games`
   - 현재 보드 상태, 턴 플레이어, 선공, 왕 침투 상태, 승자, 종료 사유 관리
3. `moves`
   - 이동 또는 포로 배치 로그를 append-only로 저장
4. `player_stats`
   - 십이장기 전용 승/패 캐시
5. 상태 스냅샷은 JSONB로 저장한다.
   - `board`: 각 칸의 말, 소유자, 승격 여부
   - `hands`: 각 플레이어 포로 목록
   - `turn`: 현재 턴 플레이어
   - `pending_try_owner`: 왕 침투 승리 판정 대기 플레이어
6. 수순 로그는 복구, 리플레이, 디버깅 용도로 별도 보관한다.

## 서버 작업 단계
1. 규칙 기반 상태 머신을 먼저 고정한다.
   - 이동 가능 좌표 계산
   - 잡기 처리
   - 승격 처리
   - 포로 배치 가능 칸 검증
   - 왕 잡기 승리 판정
   - 왕 침투 승리 판정
2. 전용 SQL 문서를 추가한다.
   - 예시 파일명: `docs/supabase-twelve-janggi.sql`
3. RPC를 게임 흐름별로 분리한다.
   - `create_room`
   - `join_room`
   - `set_guest_ready`
   - `start_game`
   - `move_piece`
   - `drop_piece`
   - `leave_room`
   - `reset_room`
   - `cleanup_stale_finished_rooms`
4. 클라이언트는 RPC만 호출하고, 말 이동 합법성의 최종 판정은 DB 함수가 담당한다.
5. Realtime 구독은 룸 메타 테이블과 게임 상태 테이블만 본다.

## 프론트엔드 작업 단계
1. 새 라우트를 추가한다.
   - `/games/twelve-janggi`
2. 보드게임 목록에서 `lib/board-games.ts`에 있는 `십이장기` 항목을 실제 진입 가능 상태로 전환한다.
3. 페이지 구조는 흑과백과 동일한 3단 흐름으로 맞춘다.
   - 로비 진입 전 인증 확인
   - 대기실
   - 실시간 플레이 화면
4. 게임 컴포넌트는 흑과백처럼 단일 페이지 흐름을 유지하되, 내부 상태는 십이장기 전용 타입으로 새로 만든다.
5. 색상 시스템은 녹색 계열 토큰으로 분리한다.
   - 배경: 짙은 녹청/올리브
   - 강조: 연두/비취
   - 경고: 흑과백의 적색 계열과 구분되는 황록 계열
6. 방 생성, 입장 코드 입력, 준비 버튼, 시작 버튼, 나가기 버튼의 위치와 위계는 흑과백 UX를 따라간다.
7. 보드는 모바일과 데스크톱 모두에서 한눈에 3x4가 읽히도록 만든다.
8. 말 선택 시 이동 가능 칸, 잡기 가능 칸, 포로 배치 가능 칸을 명확히 시각화한다.
9. 각 진영과 포로 영역은 소유자 방향이 헷갈리지 않도록 항상 고정 레이아웃으로 표시한다.

## 우선 구현 범위
1. 1:1 실시간 대전
2. 방 생성, 코드 입장, 준비, 시작, 나가기
3. 기본 이동과 포로 배치
4. 승격과 포로 재사용
5. 왕 포획 승리와 왕 침투 승리
6. 종료 후 보드 상태와 수순 로그 확인

## 보류 또는 확인 필요 항목
1. 보드 크기 표기 충돌
   - 사용자 원문은 `가로 4칸, 세로 3칸`
   - 같은 원문에서 `각 진영 3칸`
   - 현재 기준안은 `3열 x 4행`
2. 선공 결정 방식
   - 기준안: 게임 시작 시 랜덤
3. 반복 수나 장기전 무승부 처리
   - 현재는 정의 없음
4. 전적 표기를 흑과백과 완전히 분리할지, 랜딩에서 게임별 전적을 따로 보여줄지 여부

## 구현 순서 제안
1. 규칙 문서 확정
2. Supabase 전용 스키마 또는 접두사 전략 확정
3. SQL 상태 머신과 RPC 작성
4. 십이장기 전용 타입과 클라이언트 데이터 로더 작성
5. 대기실 UI 이식
6. 보드/말/포로 UI 구현
7. Realtime 동기화와 종료 처리 연결
8. 모바일 레이아웃과 예외 흐름 검증
