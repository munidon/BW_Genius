# 러브 레터 구현 계획

## 목표
1. 같은 사이트 안에 `러브 레터`를 새 보드게임으로 추가한다.
2. 방 생성 시 `2인`, `3인`, `4인` 중 하나를 반드시 선택해야만 방을 만들 수 있게 한다.
3. 로비, 대기실, 종료 흐름은 기존 보드게임 UI 지침을 따르되, 플레이 화면은 `러브 레터` 전용의 다인용 카드 테이블 UX로 구현한다.
4. 카드 드로우, 공개, 지목, 교환, 탈락, `비밀 폴라로이드` 토큰 획득 연출을 모두 동적으로 보여 주는 것을 v1 요구사항으로 둔다.

## 가이드라인 적용과 예외
1. 아래 공통 흐름은 `docs/guidelines/boardgame-ui-guidelines.md`를 그대로 따른다.
   - 인증/닉네임 가드
   - `Room Lobby`
   - `waiting / playing / finished` 3상태
   - 플레이 중 `기권 후 나가기`
   - 종료 오버레이와 `Room으로 복귀` 동선
2. 입장 전 `Room Lobby` 하단에는 러브 레터 핵심 규칙 요약 카드를 둔다.
   - 목적: 목표 `비밀 폴라로이드` 토큰 수, 턴 진행 방식, 주요 카드 역할, 라운드 승리 조건을 빠르게 이해시키기 위함
3. 대기실 상단 구조는 기존과 같은 정보 밀도를 유지한다.
   - 좌측: `ROOM CODE`와 복사 버튼
   - 우측: `STATUS`
   - 하단: 참가자 카드 목록
   - 최하단: 준비/시작/나가기 버튼
4. 기존 지침의 `호스트/게스트 2열 카드`는 러브 레터에서만 `참가자 카드 그리드`로 확장한다.
5. 다만 참가자 카드 그리드에서도 `호스트는 준비하지 않고 상태 대기만 표시`, `비호스트 참가자만 준비 상태를 표시` 규칙은 유지한다.
6. 이 예외가 필요한 이유는 러브 레터가 2~4인 방을 지원하기 때문이다.
7. 플레이 화면에서는 항상 `내 좌석`을 하단 중앙에 고정하고, 상대 좌석은 인원수에 맞춰 회전 배치한다.
   - 2인: 정면
   - 3인: 삼각형
   - 4인: 사각형

## 핵심 규칙 기준
1. 사용자 제공 규칙과 `docs/game-rules/love-letter-rules.md`를 구현 기준으로 사용한다.
2. 덱은 21장 고정 구성이다.
3. 첫 라운드 선 플레이어는 무작위다.
4. 첫 라운드 이후에는 직전 라운드 승자가 선 플레이어다.
5. 2인전도 21장 전체를 사용하고, 비공개 제거 카드 1장만 둔다.
6. 승리 목표 `비밀 폴라로이드` 토큰 수는 인원수별로 다르게 적용한다.
   - 2인: 7개
   - 3인: 5개
   - 4인: 4개
7. 이 게임에서는 `사용`과 `버림`을 같은 공개 행동으로 간주하고, 모두 자기 앞 공개 카드 더미에 누적한다.
8. 덱이 바닥난 라운드에서는 손패 숫자 우선, 자기 앞 공개 카드 숫자 합 차선 순위로 승자를 판정한다.
9. 자기 앞 공개 카드 숫자 합까지 같으면 공동 라운드 승자로 처리하고, 다음 라운드 선 플레이어는 그들 중 무작위로 정한다.
10. 라운드 종료 후 최종 승리 조건이 아직 충족되지 않았다면, 호스트가 `다음 라운드` 버튼을 눌러야 다음 라운드로 진행한다.
11. 플레이 중 참가자 누구든 `기권 후 나가기`를 하거나 `5분` 동안 서버가 감지한 활동이 없어 잠수 판정이 나면, 남아 있는 모든 플레이어를 즉시 승리 처리하고 종료 안내 후 `Room 나가기`만 제공한다.

## 분리 전략
1. 인증과 공용 닉네임은 기존 `auth.users`와 현재 프로필 흐름을 그대로 사용한다.
2. 러브 레터 데이터는 기존 `bw_*`, `tj_*` 구조와 분리해 `ll_*` 접두사로 관리한다.
3. 러브 레터는 다인용 게임이므로, 기존 1:1용 `host_id / guest_id` 중심 모델을 재사용하지 않는다.
4. 룸 메타, 참가자 목록, 공개 로그, 전적, 라운드 상태는 러브 레터 전용 테이블과 RPC로 분리한다.

## 권장 데이터 구조
1. `ll_rooms`
   - 방 코드
   - 호스트 ID
   - `player_limit`
   - `waiting / playing / finished` 상태
   - 현재 라운드 번호
   - 목표 `비밀 폴라로이드` 토큰 수
   - 최종 승자 목록
   - 마지막 이탈 또는 잠수 플레이어 닉네임
   - 갱신 시각
2. `ll_room_players`
   - 방 ID
   - 플레이어 ID
   - 좌석 인덱스
   - 참가 순서
   - 준비 상태
   - 누적 `비밀 폴라로이드` 토큰 수
   - 닉네임 스냅샷
   - 방 이탈 여부
   - 마지막 활동 시각
3. `ll_round_states`
   - 방 ID
   - 라운드 번호
   - 선 플레이어 ID
   - 현재 턴 플레이어 ID
   - 비공개 제거 카드
   - 덱 순서
   - 플레이어별 손패
   - 플레이어별 자기 앞 공개 카드 더미
   - 보호 상태
   - 탈락 상태
   - 관전자 권한 플레이어 목록
   - `Broadcaster(6)` 대기 중 여부
   - 현재 라운드 승자 목록
   - 종료 사유
   - 공개 결과 잠금 여부
   - 관전자 권한 회수 완료 여부
4. `ll_action_logs`
   - 공개 가능한 수순 로그를 append-only로 저장한다.
   - 예시: 드로우, 카드 공개, 지목, 탈락, 라운드 종료, `비밀 폴라로이드` 토큰 획득
5. `ll_player_stats`
   - 게임별 누적 전적
   - 매치 수
   - 우승 수
   - 라운드 승리 수
6. 민감한 손패 정보는 직접 테이블 조회로 노출하지 않고, RPC를 통해 사용자별로 가공한 뷰만 반환한다.

## 공개 정보와 비공개 정보 처리 전략
1. 러브 레터는 `현재 손패`가 비공개라서, 십이장기나 흑과백처럼 단순 Realtime 테이블 노출만으로는 부족하다.
2. 서버는 `ll_round_states`에 전체 상태를 권한 있게 저장한다.
3. 클라이언트는 공개 테이블 구독과 별도로 `ll_get_room_view(room_id)` RPC를 호출해 개인화된 상태를 받는다.
4. `ll_get_room_view`는 아래 정보만 현재 사용자 기준으로 노출한다.
   - 내 손패
   - 내가 선택해야 하는 추가 입력 상태
   - 공개 로그
   - 자기 앞 공개 카드
   - 보호/탈락/`비밀 폴라로이드` 토큰 수
   - 남은 덱 장수
   - 현재 턴과 선 플레이어
   - 라운드 종료 시 동점 판정용 자기 앞 공개 카드 합산 정보
   - 라운드 종료 공개 상태 여부
   - 라운드 승자 정보
5. 단, 현재 라운드에서 이미 탈락한 모든 플레이어에게는 같은 관전자 모드 뷰를 반환한다.
   - 남아 있는 모든 플레이어의 현재 손패
   - 관전자 모드 배너 상태
   - 본인은 행동 불가라는 안내
6. 라운드 종료 공개 상태에서는 탈락 여부와 무관하게 모든 플레이어에게 모든 손패와 라운드 승자 정보를 함께 노출한다.
7. 호스트가 `다음 라운드`를 눌러 새 라운드가 시작되면, 관전자 권한과 라운드 종료 공개 상태를 즉시 회수한다.
8. 새 라운드는 `관전자 권한 회수 완료 여부`가 확인된 뒤에만 시작한다.
9. `Counselor(2)`와 `Broadcaster(6)`의 비공개 결과는 RPC 응답으로만 돌려주고, 공개 로그에는 결과의 범위만 남긴다.
10. `Athlete(3)` 카드 비교 숫자는 비교 당사자에게만 보이고, 다른 플레이어에게는 탈락 여부만 공개한다.
11. Realtime 구독은 공개 메타 변경 감지용으로 쓰고, 실제 화면 데이터는 감지 후 `ll_get_room_view`를 다시 받아 회수한다.
12. 단, 누군가 잠수 또는 기권으로 전체 게임 종료가 발생하면 관전자 권한과 라운드 공개 상태보다 종료 전환을 우선한다.

## 상태 머신 기준
1. 룸 상태는 아래 3가지를 유지한다.
   - `waiting`
   - `playing`
   - `finished`
2. 라운드 내부 상태는 최소한 아래 단계를 가진다.
   - `dealing`
   - `await_turn`
   - `await_broadcaster_resolution`
   - `round_reveal`
   - `await_next_round`
   - `match_finished`
3. 일반 카드 사용은 가능한 한 `한 번의 RPC`로 끝낸다.
4. 예외는 `Broadcaster(6)`다.
   - 1차 RPC: 카드 사용 선언, 추가 카드 2장 확인
   - 2차 RPC: 어떤 카드를 남기고 어떤 순서로 덱 아래에 둘지 제출
5. `Scholar(8)` 강제 버림, `Crush(9)` 즉시 탈락, `Newbie(0)` 보너스 판정은 모두 서버 상태 머신에서 처리한다.
6. 라운드 종료 시 승자 판정 순서는 서버에서 강제한다.
   - 생존자 1명 여부
   - 손패 숫자 비교
   - 자기 앞 공개 카드 숫자 합 비교
   - 완전 동률 처리
7. 라운드 종료 시 서버는 즉시 모든 손패를 공개 가능한 `round_reveal` 상태로 전환한다.
8. 최종 승리 조건이 아직 충족되지 않았다면, 방 호스트만 `다음 라운드`를 실행할 수 있다.
9. `다음 라운드` 실행 시 서버는 아래 순서를 강제한다.
   - 관전자 권한 회수 요청
   - 모든 플레이어의 관전자 권한 회수 완료 확인
   - 공개 결과 상태 종료
   - 새 덱 셔플과 배분
   - 새 라운드 시작
10. 최종 승리 조건이 충족되면 `await_next_round`를 거치지 않고 바로 `match_finished`로 전환한다.
11. 참가자 누구든 `기권 후 나가기`를 하거나 `5분 무활동` 잠수가 감지되면, 해당 플레이어를 패배 처리하고 남은 모든 플레이어를 승자로 기록한 뒤 `finished` 상태에 진입한다.

## 권장 RPC 흐름
1. `ll_create_room(player_limit, nickname_snapshot)`
   - `player_limit`은 `2 / 3 / 4`만 허용한다.
2. `ll_join_room(room_code, nickname_snapshot)`
3. `ll_set_player_ready(room_id, ready)`
   - 호스트를 제외한 참가자만 호출한다.
4. `ll_start_match(room_id)`
   - 선택한 정원만큼 플레이어가 모두 입장했고, 호스트를 제외한 참가자가 모두 준비된 상태에서만 시작한다.
5. `ll_get_room_view(room_id)`
6. `ll_play_card(room_id, played_card, target_player_id, guessed_card)`
   - `Monitor`, `Counselor`, `Athlete`, `VicePres`, `President` 같은 카드까지 처리한다.
7. `ll_resolve_broadcaster(room_id, kept_card, bottom_order)`
8. `ll_advance_to_next_round(room_id)`
   - 호스트만 호출 가능하다.
   - 최종 승리 조건이 만족되지 않은 `await_next_round` 상태에서만 호출 가능하다.
   - 관전자 권한 회수 완료가 확인되어야 성공한다.
9. `ll_leave_room(room_id)`
   - `playing` 상태에서는 `기권 후 나가기`와 동일하게 즉시 전체 게임 종료를 트리거한다.
10. `ll_reset_room(room_id)`
11. `ll_cleanup_stale_finished_rooms()`

## 서버 작업 단계
1. 러브 레터 전용 타입과 상태 모델을 먼저 고정한다.
   - 카드 ID/이름/장수
   - 플레이어별 손패
   - 라운드 종료 판정
   - `비밀 폴라로이드` 토큰 누적
2. 서버 권한에서 셔플과 라운드 세팅을 구현한다.
3. 카드 효과별 판정 함수를 만든다.
   - 지목 가능 대상 계산
   - 보호 상태 확인
   - 탈락 처리
   - 탈락 직후 관전자 모드 전환
   - 라운드 종료 후 전체 공개 상태 전환
   - `다음 라운드` 실행 시 관전자 권한 회수
   - 관전자 권한 회수 완료 확인 후 다음 라운드 시작
   - 참가자 이탈 또는 참가자 `5분 무활동` 종료 처리
   - 마지막 활동 시각 갱신
   - 손패 교환
   - 강제 버림과 재드로우
   - `Scholar` 자동 버림
   - `Crush` 즉시 탈락
   - `Newbie` 보너스 `비밀 폴라로이드` 토큰
   - 덱 소진 시 손패 숫자 비교
   - 동점 시 자기 앞 공개 카드 합 비교
4. 러브 레터 전용 SQL 문서를 추가한다.
   - 예시 파일명: `docs/schemas/ll-schema.sql`
5. RLS는 `같은 방 참가자만 공개 정보에 접근 가능`하도록 제한한다.
6. 비공개 손패는 직접 `select`가 아니라 RPC를 통해서만 받게 만든다.
7. Realtime 채널과 보조 폴링을 둘 다 붙여 종료 상태 누락을 회수한다.

## 프론트엔드 구조 제안
1. 새 라우트를 추가한다.
   - `/games/love-letter`
2. 전용 컴포넌트와 유틸을 만든다.
   - `components/love-letter-page-client.tsx`
   - `components/love-letter-online.tsx`
   - `lib/love-letter.ts`
3. `lib/board-games.ts`의 `러브 레터` 항목은 구현 완료 후 `playable` 상태로 전환한다.
4. 게임 상태 타입은 흑과백, 십이장기 타입과 분리한다.
5. 러브 레터 전용 테마 토큰을 만든다.
   - 배경: 짙은 남색 + 와인빛
   - 강조: 크림, 코랄, 금빛 폴라로이드 하이라이트
   - 성공/보호: 민트 또는 라임 계열
   - 위험/탈락: 붉은 체리 계열
6. 이미 존재하는 카드 일러스트를 우선 재사용한다.
   - `public/images/love_letters/*.png`

## Room Lobby / 대기실 구현
1. 입장 전 `Room Lobby`는 기존 게임과 같은 2열 구조를 유지한다.
2. `방 만들기` 버튼을 누르기 전 `2인 / 3인 / 4인` 선택 UI를 반드시 통과하게 한다.
3. 인원 선택 전에는 방 생성 버튼을 비활성화한다.
4. 방 코드 입장 UI는 기존 게임과 같은 위치에 둔다.
5. 대기실 참가자 카드는 아래 정보를 같은 밀도로 보여 준다.
   - 역할
   - 닉네임
   - 준비 상태 (`호스트는 상태 대기`, `비호스트 참가자는 상태 준비 완료 / 상태 미준비`)
   - 현재 누적 `비밀 폴라로이드` 토큰 수
6. 호스트는 정원이 꽉 차고 비호스트 참가자가 모두 준비된 뒤에만 `게임 시작` 버튼을 누를 수 있다.

## 플레이 화면 구조
1. 상단에는 현재 라운드, 선 플레이어, 현재 턴, 남은 덱 수, 목표 `비밀 폴라로이드` 토큰 수를 둔다.
2. 중앙 테이블에는 아래 요소를 둔다.
   - 드로우 덱
   - 비공개 제거 카드 자리 표시
   - 공개 로그 또는 최근 행동 피드
   - 라운드 종료 안내
   - 동점 시 비교되는 자기 앞 공개 카드 숫자 합 요약
3. 플레이어 좌석은 인원수별로 다르게 배치한다.
   - 2인: 상대 1명 상단 중앙
   - 3인: 상대 2명 좌상단, 우상단
   - 4인: 상대 3명 좌상단, 상단 중앙, 우상단 또는 정사각형 회전 배치
4. 내 손패 영역은 하단 도크에 고정한다.
5. 내 차례에는 손패 2장을 나란히 강조하고, 지금 낼 수 있는 카드와 낼 수 없는 카드를 명확히 구분한다.
6. 각 상대 좌석에는 아래 공개 정보를 둔다.
   - 플레이어 이름
   - 남은 손패 장수
   - 자기 앞 공개 카드 더미
   - 현재 자기 앞 공개 카드 숫자 합
   - 보호 상태
   - 탈락 상태
   - 누적 `비밀 폴라로이드` 토큰 수
7. 모든 플레이어는 화면 상단 공용 점수판에서 `비밀 폴라로이드` 토큰 트랙을 본다.
   - 토큰 이미지는 별도 리소스를 사용한다.
   - 인원수별 목표 개수만큼 빈 슬롯을 먼저 그린다.
   - 토큰을 얻을 때마다 슬롯이 하나씩 채워진다.
   - 이 변화는 모든 플레이어 화면에 동시에 보여야 한다.
8. 내가 라운드에서 탈락하면 화면을 관전자 모드로 전환한다.
   - 탈락이 확정되는 즉시 내 입력을 잠그고 손패 선택 UI를 비활성화
   - 중앙 안내 문구 `해당 라운드에서 탈락되었습니다. 관전자 모드가 실행됩니다.` 를 잠시 출력
   - 안내 문구가 사라진 뒤 관전자 모드 레이아웃으로 전환
   - 남아 있는 플레이어 손패를 모두 공개
   - `라운드 종료까지 관전 중` 안내 배너 노출
9. 라운드가 종료되면 화면을 공개 결과 상태로 전환한다.
   - 모든 플레이어 손패를 동시에 공개
   - 라운드 승자 또는 공동 승자를 중앙 결과 카드로 표시
   - 라운드 승자는 초상화 영역에 노란색 테두리로 강조
   - 공동 라운드 승리라면 해당 플레이어들을 모두 같은 방식으로 강조
   - 최종 승리 조건이 아직 미충족이면 호스트에게만 `다음 라운드` 버튼 노출
   - 최종 승리 조건이 충족되면 `다음 라운드` 대신 최종 승리 화면으로 전환
   - 호스트가 버튼을 누르기 전까지 현재 공개 결과 상태를 유지
10. 최종 승리 화면에서는 최종 승자 초상화에 왕관이 있는 노란색 테두리를 적용한다.
   - 공동 최종 승리라면 모든 최종 승자를 같은 방식으로 강조한다.
11. 참가자 누구든 이탈하거나 `5분` 동안 서버가 감지한 활동이 없어 잠수 판정이 나면 남은 플레이어 화면을 종료 안내 상태로 전환한다.
   - 문구: `@@님이 떠나 게임을 종료합니다`
   - 버튼: `Room 나가기`
   - 다른 조작은 모두 비활성화

## 카드 효과별 UI/UX 계획
1. `Monitor(1)`
   - 카드 선택 후 타겟 선택 오버레이
   - 이어서 추측 가능한 카드 목록 모달
   - `Monitor` 자체는 선택지에서 비활성화
2. `Counselor(2)`
   - 타겟 선택 후 내 화면에만 손패 공개 모달
3. `Athlete(3)`
   - 타겟 선택 후 두 카드가 중앙으로 슬라이드
   - 비교 숫자는 당사자에게만 보이고, 다른 사람에게는 결과 텍스트만 표시
4. `Librarian(4)`
   - 보호 상태 동안 좌석 카드에 실드 오라와 아이콘 유지
5. `VicePres(5)`
   - 버림 애니메이션 후 새 카드가 덱에서 날아와 손패로 들어오는 연출
   - 비공개 제거 카드를 뽑는 경우 별도 툴팁 표시
6. `Broadcaster(6)`
   - 카드 3장을 펼쳐 보여 주는 전용 모달
   - 남길 카드 1장 선택
   - 남은 2장은 클릭 또는 드래그로 순서 지정
   - 도움말 문구: `고르는 순서대로 덱 맨 아래로 들어갑니다`
   - 제출 후 선택한 순서대로 카드가 덱 아래로 들어가는 애니메이션
7. `President(7)`
   - 두 플레이어 사이에 카드가 교차 이동하는 애니메이션
8. `Scholar(8)`
   - 강제 버림 시 자동 공개와 짧은 경고 토스트
9. `Crush(9)`
   - 공개와 동시에 탈락 연출
   - 탈락 플레이어 화면에는 `해당 라운드에서 탈락되었습니다. 관전자 모드가 실행됩니다.` 문구를 먼저 짧게 노출
10. `Newbie(0)`
   - 라운드 종료 후 보너스 `비밀 폴라로이드` 토큰 획득 시 별도 보너스 팝업
11. `라운드 종료`
   - 모든 손패 공개
   - 승자 하이라이트
   - 호스트 전용 `다음 라운드` 버튼 노출
12. `참가자 이탈 종료`
   - `@@님이 떠나 게임을 종료합니다` 문구 표시
   - `Room 나가기` 버튼만 노출

## 애니메이션 우선순위
1. 라운드 시작 시 카드 배분 애니메이션
2. 현재 턴 플레이어 하이라이트와 턴 배턴 이동
3. 드로우 덱에서 손패로 이어지는 카드 이동
4. 카드 사용 시 중앙 공개 연출
5. 타겟 지목 시 선택 가능한 상대 하이라이트
6. 탈락 시 좌석 카드 desaturate + 퇴장 모션
7. 탈락 직후 중앙 문구 `해당 라운드에서 탈락되었습니다. 관전자 모드가 실행됩니다.` 표시
8. 관전자 모드 진입 시 카메라/레이아웃 전환과 관전 배너 등장 연출
9. 라운드 종료 시 모든 손패 공개와 승자 하이라이트 연출
10. `비밀 폴라로이드` 토큰 획득 시 토큰 슬롯 채움 연출
11. 호스트의 `다음 라운드` 실행 시 관전자 권한 회수 완료 확인 후 공개 상태가 닫히고 새 라운드 카드 배분으로 전환
12. 최종 승리 화면에서 왕관이 있는 노란색 테두리 강조
13. 참가자 이탈 종료 시 종료 문구와 `Room 나가기` 버튼 전환
14. 게임 종료 시 중앙 오버레이와 최종 타임라인 요약

## 모바일 대응 기준
1. 모바일에서도 `내 손패`는 항상 하단에 고정한다.
2. 상대 좌석은 작은 아바타 카드로 압축하되, `삼각형 / 사각형` 관계는 유지한다.
3. 자기 앞 공개 카드가 길어지면 최근 카드 우선으로 보이고, 전체 목록은 펼침 패널에서 확인하게 한다.
4. 타겟 선택과 카드 설명은 바텀 시트 형태를 우선 검토한다.
5. `Broadcaster(6)` 전용 모달은 모바일에서 세로 스택 또는 가로 스와이프 카드로 대응한다.
