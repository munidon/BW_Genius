# 보드게임 UI 통일 지침서

## 목적
이 문서는 같은 사이트 안에 새 보드게임을 추가할 때, `흑과백`과 `십이장기`에서 맞춘 공통 UX 규칙을 유지하기 위한 기준 문서다.

## 기준 레퍼런스
1. 대표 레퍼런스 구현
   - `components/black-white-online.tsx`
   - `components/twelve-janggi-online.tsx`
2. 새 게임은 위 두 구현의 공통 구조를 따르고, 게임 고유 규칙과 시각 테마만 바꾼다.

## 공통 페이지 구조
1. 상단 헤더
   - 게임명
   - 짧은 설명
   - 내 닉네임
   - 내 전적
   - 랜딩으로 돌아가는 버튼
2. 인증/닉네임 가드
   - 비로그인 상태 안내
   - 닉네임 미설정 안내
3. `Room Lobby`
   - 방 만들기
   - 6자리 방 코드 입력
   - 입장 버튼
4. 방 상태 영역
   - ROOM CODE + 복사 버튼
   - STATUS
   - 호스트/게스트 카드
   - 준비/시작/나가기 버튼
5. 방 코드 노출 규칙
   - `waiting` 상태에서만 `ROOM CODE`와 복사 버튼을 노출한다.
   - 대국이 시작된 뒤(`playing`, `finished`)에는 방 코드 블록을 숨긴다.
6. 플레이 영역
   - 현재 턴/진행 상태
   - 게임 핵심 인터랙션 영역
   - 보조 정보 또는 로그
7. 종료 영역
   - 게임 종료 요약
   - 복귀/나가기 버튼
   - 종료 사유 안내
8. 입장 전 `Room Lobby` 하단 규칙 요약 또는 게임 안내 카드

## Room Lobby UI 규칙
1. 입장 전 로비 카드는 흑과백 구조를 기본으로 사용한다.
   - 상단 제목: `Room Lobby`
   - 본문 2열 구성: `방 만들기` 버튼 / `방 코드 입력 + 입장 버튼`
2. 입장 전 `Room Lobby` 바로 아래에는 게임 규칙 안내 섹션을 둔다.
   - 목적: 방 생성/입장 전에 핵심 규칙을 바로 확인시키기 위함
   - 최소 내용: 판/말 배치, 턴 진행 방식, 승리 조건
   - 형식: 2~4개의 요약 카드 또는 동일 밀도의 안내 블록
3. 대기실 상단 카드는 아래 순서를 유지한다.
   - 좌측: `ROOM CODE`와 바로 옆 `복사` 버튼
   - 우측: `STATUS`
   - 하단: 호스트/게스트 카드 2열
   - 최하단: 준비/시작/나가기 버튼
4. 호스트/게스트 카드는 같은 밀도와 같은 정보 구조를 유지한다.
   - 역할명
   - 닉네임
   - 현재 상태
   - 전적
5. `waiting` 상태의 플레이어 카드는 점수 대신 상태 문구를 우선 노출한다.
   - 호스트: `상태 대기`
   - 게스트: `상태 준비 완료` 또는 `상태 미준비`
   - 점수는 `playing` 또는 `finished` 요약 영역에서만 노출한다.
6. 대기실 액션 버튼은 아래 순서를 유지한다.
   - 게스트: `준비` 또는 `준비 취소`
   - 호스트: `게임 시작`
   - 공통: `Room 나가기`
7. `waiting` 상태의 Room Lobby는 게임별 테마 색만 다르고, 카드 구조와 버튼 위계는 동일해야 한다.

## 상태 흐름 통일 규칙
1. 모든 게임은 최소한 다음 3상태를 가진다.
   - `waiting`
   - `playing`
   - `finished`
2. `waiting`
   - 방 생성/입장
   - 게스트 준비
   - 호스트 시작
3. `playing`
   - 현재 턴/행동 가능 여부 노출
   - `기권 후 나가기` 버튼 노출
4. `finished`
   - 승패 요약 카드 노출
   - 종료 오버레이 연출
   - `Room으로 복귀` 또는 `게임 나가기` 동선 제공

## 시작 연출 규칙
1. 1:1 대전 게임은 `waiting -> playing` 전환 시 코인 플립 애니메이션 화면을 띄운다.
2. 시작 연출은 각 플레이어에게 자신이 `선 플레이어`인지 `후 플레이어`인지 바로 알려줘야 한다.

## 종료 화면 규칙
1. 종료 시 중앙 승패 오버레이를 띄운다.
   - 승리 시: `승리!`
   - 패배 시: `패배...`
   - 무승부 게임이라면 별도 무승부 텍스트 사용 가능
2. 종료 카드에는 최소한 아래 정보를 포함한다.
   - `게임 종료` 제목
   - 최종 승자 또는 결과
   - 종료 사유
   - 게임별 핵심 요약 정보
3. 종료 후 버튼 순서는 다음을 기본으로 한다.
   - 호스트: `Room으로 복귀`
   - 공통: `게임 나가기`
4. 종료 화면에서는 플레이 중 컨트롤보다 종료 정보가 우선 시각 요소가 되어야 한다.

## 기권 처리 규칙
1. 플레이 중 이탈은 즉시 기권 처리한다.
2. 상대가 기권하면 승리 플레이어는 종료 화면으로 반드시 진입해야 한다.
3. 기권 종료 시 안내 문구를 보여준다.
   - 예시: `상대 플레이어가 기권했습니다.`
4. 기권 종료 후 상대가 이미 나간 상태라면 `Room으로 복귀` 버튼은 숨긴다.
5. 이 경우 다음 안내 문구를 함께 보여준다.
   - 예시: `상대가 기권 후 나가 Room으로 복귀할 수 없습니다.`

## 버튼 배치 규칙
1. 대기실
   - 게스트: `준비` 또는 `준비 취소`
   - 호스트: `게임 시작`
   - 공통: `방 나가기`
2. 플레이 중
   - 공통: `기권 후 나가기`
3. 종료 후
   - 호스트: `Room으로 복귀`
   - 공통: `게임 나가기`
4. 버튼 라벨은 게임이 달라도 동일 문구를 우선 사용한다.

## 확인 모달 규칙
1. 나가기 계열 버튼은 확인 모달을 둔다.
2. 플레이 중 나가기는 반드시 패널티를 명시한다.
   - 예시: `지금 나가면 즉시 기권 처리되고 상대 승리로 게임이 종료됩니다.`
3. 대기실 나가기는 방 이탈 안내만 간단히 표시한다.

## 실시간 동기화 규칙
1. Realtime 구독이 있어도 보조 폴링을 유지한다.
2. 특히 `playing -> finished` 전환은 Realtime 누락 시에도 회수할 수 있어야 한다.
3. 종료 감지 후 다음 후처리를 공통으로 둔다.
   - 승패 오버레이 표시
   - 기권 승리 안내 notice
   - 종료 화면으로 화면 상태 전환

## 선택/강조 규칙
1. 선택된 대상은 항상 한눈에 보여야 한다.
2. 선택된 말/카드/포로/칸은 흰색 계열 하이라이트를 우선 사용한다.
3. 행동 가능한 대상은 테마 컬러 보조 강조를 사용한다.
4. 선택 상태와 행동 가능 상태는 색상 의미가 달라야 한다.
   - 선택됨: 흰색
   - 이동/배치 가능: 테마 강조색

## 색상과 테마 규칙
1. 레이아웃 구조는 통일하되, 게임별 메인 색상은 바꿔도 된다.
2. 허용되는 변경 범위
   - 헤더 분위기
   - 포인트 컬러
   - 보드/말 스타일
   - 아이콘/배경 질감
3. 유지해야 하는 공통 범위
   - 카드 모서리 스타일
   - 버튼 위계
   - 상태 칩 위치
   - 정보 카드 밀도
   - 종료 화면 구조

## 문구 규칙
1. 상태 문구는 짧고 직관적으로 유지한다.
2. 같은 의미의 행동은 게임이 달라도 같은 표현을 우선 사용한다.
   - `Room Lobby`
   - `방 나가기`
   - `기권 후 나가기`
   - `게임 종료`
   - `Room으로 복귀`
   - `게임 나가기`
3. 시스템 안내 문구는 감탄보다 정보 전달을 우선한다.

## 새 게임 추가 체크리스트
1. `waiting / playing / finished` 3상태가 있는가
2. 인증/닉네임 가드가 있는가
3. `Room Lobby` 구조가 기존 게임과 같은가
4. 입장 전 `Room Lobby` 하단에 게임 규칙 안내 카드가 있는가
5. 플레이 중 `기권 후 나가기`가 있는가
6. 종료 시 중앙 승패 오버레이가 있는가
7. 기권 종료 시 `Room으로 복귀` 제한이 반영되는가
8. 종료 카드에 결과, 종료 사유, 요약 정보가 보이는가
9. 선택 상태가 흰색 하이라이트로 명확히 보이는가
10. Realtime 누락 시 종료 상태를 회수할 보조 동기화가 있는가
11. 게임 고유 색상만 다르고 정보 구조는 기존 게임과 통일되어 있는가

## 적용 원칙
1. 새 게임은 먼저 이 문서를 기준으로 UI 뼈대를 만든다.
2. 이후 게임 규칙에 맞춰 인터랙션만 덧붙인다.
3. 공통 구조를 깨야 한다면, 이유를 구현 계획 문서에 먼저 기록한다.
